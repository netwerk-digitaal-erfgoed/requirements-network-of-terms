Abstract

This document describes the requirements that the Network of Terms must meet in order to be used in the Dutch Digital Heritage Network (NDE).

1. Introduction

How can users find heritage information, regardless of the institutions that manage this information? This is only possible if institutions use the same terms to describe the same information. These terms come from common terminology sources such as thesauri and reference lists.

Nevertheless, it can be a challenge for institutions to use terminology sources: sources have different ways of making their terms accessible, for example with regard to search options, metadata schemas and interfaces. This variety makes them difficult to use. The Network of Terms wants to provide a solution for this by offering a uniform way of searching in terminology sources, both functionally and technically.

The Network of Terms is a generic facility, a facility that supports the network of digital heritage. The Network of Terms also is a distributed facility, a facility that is not centrally provided by one organization, but decentralized by various organizations. The "Network of Terms" is therefore an umbrella term to designate all instances â all "networks of terms".

This document specifies the functional and non-functional requirements that the Network of Terms must meet. The purpose of the requirements is to make the Network of Terms as useful as possible for users, usually collection managers of institutions and suppliers of their collection management software.

The requirements have been drawn up by the project Standardized terms in the network. The project is part of the 2019-2020 intensification program of the Dutch Digital Heritage Network. The requirements take the Digital Heritage Reference Architecture 3.0 as a starting point. The requirements are a deepening of DERA and can supplement a future version of it.

Include a glossary of terms and their definitions. This is preferably a separate document that can also be used by other documents, such as the Requirements for Datasets.

1.1. Audience

The requirements are intended for everyone involved in implementing and making available the Network of Terms, such as software developers and application managers.

1.2. Status

This document is a working draft.

2. Prioritization of requirements

Each requirement has a priority. The priority follows the MoSCoW method:

Priority Explanation
Must have The requirement is necessary
Should have The requirement is important, but not necessary
Could have The requirement is desired, but not important
Would have The requirement is interesting, but not desired now

3. Requirements

The requirements are both functional and non-functional. The functional requirements make clear which functionality the Network of Terms must offer in order to be used by users and application managers. The non-functional requirements describe how the Network of Terms must fulfill the functional requirements. The non-functional requirements are primarily intended for the IT specialists who will build the Network of Terms and the IT specialists who will implement the Network of Terms in systems of users, such as collection management systems.

3.1. The Network of Terms makes terms findable via a uniform endpoint

Priority: Must have

The Network of Terms makes terms from terminology sources easy to find. Users â or more precisely: their systems â can query the Network of Terms without having to know the variety of the underlying terminology sources, such as diverging data formats or metadata schemas.

The Network of Terms will initially use GraphQL as its interface, JSON as its data format and SKOS as its metadata schema. GraphQL, JSON and SKOS are relatively simple and known to software developers, which fosters easy adoption. Other interfaces, data formats or metadata schemas may be added later.

Discussion: GraphQL is a query language, but it doesnât support search operators, such as wildcards. Which standard or best practice should the Network of Terms use for this?
Discussion: is SKOS sufficient to express all information about terms? For example, terminology source A can use metadata schema X to indicate the "status" of a term (such as a candidate term), while terminology source B can use metadata schema Y for this. SKOS has no support to describe the "status". Should a different metadata schema be used for this â or an own ontology?

A uniform endpoint means that the Network of Terms offers the same search capabilities for all terminology sources. This, however, only applies to sources that meet the Requirements for terminology sources (in Dutch). If a source doesnât support certain functionality, the Network of Terms doesnât support it either. If a source has more functionality than stipulated by the requirements, the Network of Terms cannot use it.

3.2. The Network of Terms makes terms findable via an online search environment that is accessible to users

Priority: Should have

The Network of Terms has an endpoint for querying terminology sources (see requirement 1). However, an endpoint is meant to be used by the systems of users, not directly by users. Yet users want to be able to query the Network of Terms to get an impression of the possibilities and the results, regardless of how the users' systems use it. The Network of Terms therefore has an online search environment for users. An example of such an environment is the demonstrator of the Network of Terms.

3.3. The Network of Terms searches terminology sources registered in the Registry

Priority: Must have

The Network of Terms searches terminology sources, but does not determine which sources can be used. For this purpose the Network of Terms relies on the Registry. The Registry is the generic facility of the Dutch Digital Heritage Network that keeps track of the datasets in the network, including the available terminology sources. Use of the Registry improves clarity: anyone â both man and machine â who wants to do anything with datasets can use the Registry as an entry point.

Use of the Registry does not imply that the Network of Terms is obliged to make all terminology sources in the Registry searchable. The Network of Terms can choose to use a selection of sources, depending on its purpose and target audience. This may be the case, for example, with distributed instances of the Network of Terms. For example: instance A searches only terminology sources used in the archives sector and instance B only searches terminology sources in the museums sector.

3.4. The Network of Terms publishes information about the terminology sources it can search via an endpoint

Priority: Must have

The Network of Terms makes clear to users which terminology sources it can query. Although users can consult the Registry to see what terminology sources are available (see requirement 3), the Registry is not aware of the sources used by the Network of Terms. For example: an instance of the Network of Terms can choose to make only specific sources findable.

The Network of Terms publishes the following information for each terminology source:

Property Explanation
ID The unique identifier of the terminology source, e.g. "https://data.niod.nl/WO2_Thesaurus"
Name The name of the terminology source, e.g. "Dutch World War Two Thesaurus"
Alternate name The altername name of the terminology source, if any. E.g. "WW2 Thesaurus"
Name of organization The name of the organization that manages the terminology source, e.g. "NIOD Institute for War, Holocaust and Genocide Studies"
Alternate name of organization The altername name of the organization that manages the terminology source, if any. E.g. "NIOD"
Description A description of the terminology source, e.g. "The WW2 Thesaurus is a validated, hierarchically organized keyword list for the thematic disclosure of (digital) sources from the period 1940-1945"

This information is published in all languages that the terminology source supports. This enables multilingual applications to use the information.

The Network of Terms publishes this information via an endpoint. This allows applications to process the information automatically. As with requirement 1 the endpoint will use GraphQL as its interface and JSON as its data format. For publishing information about the managing organizations Schema.org is used as metadata schema.

Discussion: is it useful to also publish this information as RDF with e.g. DCAT or Schema.org as metadata schemas? In keeping with the Requirements for Datasets, this allows applications to use the information with Linked Data technology.

3.5. The Network of Terms can search in one and more terminology sources

Priority: Must have

Some users want the Network of Terms to find terms in a particular terminology source. For example: a user may want to use the WW2 Thesaurus because it best describes the userâs heritage information. The Network of Terms should support this by having users indicate which source must be queried.

At the same time, other users want the Network of Terms to find terms in multiple terminology sources. Users want to perform these searches as efficiently as possible: they do not want to have to submit their search query multiple times to multiple sources, but once. The Network of Terms should support this by having users indicate which sources must be queried. The Network of Terms then searches these sources and returns all matching terms in one result set.

For optimal performance the Network of Terms searches terminology sources parallelly, not sequentially.

Terminology sources have different response times, for example depending on their technical infrastructure and the users' queries. Since the Network of Terms only returns a search result to the user when all sources have completed their search, the performance of the Network of Terms is determined by the source with the lowest performance. In order to not degrade the overall performance of the Network of Terms, it aborts the search in a source if, for whatever reason, the source does not respond within three seconds. In that case the search result to the user signals that the source has been skipped.

3.6. The Network of Terms queries terminology sources directly

Priority: Must have

The Network of Terms queries terminology sources directly, in real time. This guarantees that users work with current terms â the sources provide the most recent information. This also allows the Network of Terms to be lightweight: it does not have to maintain its own database with terms and does not have to seed this database periodically with terms from the terminology sources.

The consequence of this approach is that the Network of Terms fully relies on the functional and technical capabilities of the terminology sources. For instance: if a source does not offer certain functionality (e.g. full-text search) or is technically unavailable (e.g. due to a malfunction), then the source cannot be (fully) used by the Network of Terms.

3.7. The Network of Terms makes clear from which terminology sources terms originate

Priority: Must have

Users want to search in multiple terminology sources at the same time (see requirement 5). For the terms it finds, the Network of Terms must make clear from which sources they originate. This way users can make a choice. For instance: if a user searches in two sources and both sources return results, he can decide to use a term from one source, not the other.

3.8. The Network of Terms explains why no terms have been found in a terminology source

Priority: Must have

The Network of Terms queries terminology sources directly (see requirement 6). If a terminology source does not return results then the Network of Terms must make clear why this is the case. This allows users to take this into account when interpreting the results.

There are two reasons why a terminology source does not return results:

  1. A source does not contain terms that match the userâs query.

  2. A source is not available. This may be due to, for example, connection problems between the Network of Terms and the terminology source or because the terminology source takes too much time to answer.

3.9. The Network of Terms publishes specific information about terms

Priority: Must have

Users want to see specific information per found term so that they can decide whether or not to use it. This concerns the following information:

Property Explanation Required?
URI The unique identifier of the term Yes
Terminology source Name of the terminology source to which the term belongs Yes
Label The preferred label or name of the term Yes
Alternative label The alternative label or altername name of the term, e.g. a synonym No
Hidden label The hidden label of the term, e.g. a common misspelling No
Note Additional information about the scope or use of the term ("scope note") No
Related term Term that has a (non-hierarchical) relationship with the term No
Broader term Term that is hierarchically directly above the term No
Narrower term Term that is hierarchically directly underneath the term No
Status Status of the term, e.g. "candidate", "approved", "obsolete" No

The Network of Terms transforms the information of the terminology sources into the data structure above. This way the Network of Terms can provide uniform information to users, regardless of the data structure of the terminology sources.

3.10. The Network of Terms searches for terms in a specific language

Priority: Should have

Users want to be able to search for terms in a specific language. For example: one user searches in Dutch (Concentratiekampen), the other in German (Konzentrationslager) and yet another in English (Concentration camps). Therefore users must be able to submit their queries to the Network of Terms in a specific language and the Network of Terms must search for terms that match the specified language.

The language support of the Network of Terms depends on the language capabilities of the underlying terminology sources. If a user indicates that he wants to find terms in English, but a terminology source does not have English translations available for all terms, then the Network of Terms will not find these terms (even though the terms may be available in other languages).

Discussion: should the Network of Terms have a default language? In other words: if a user does not specify a language, does the Network of Terms automatically search for Dutch terms? Or should this default language be configurable? For example: a specific instance would then be able to use English as its default.

3.11. The Network of Terms automatically completes queries

Priority: Should have

Users want to search as efficiently as possible. Therefore they do not want to enter an entire query before being able to search. Instead, they want to enter part of a query. Every keystroke should then trigger a search; the search should return terms matching the exact input but also terms that start with the input. For example: this way the query Max results in the term "Max" but also in the automatically completed terms "Max Lewin" and "Maximilian of Egmont".

The support of auto-completion by the Network of Terms depends on the capabilities of the underlying terminology sources. If a terminology source does not have auto-completion, then the Network of Terms cannot provide it for that source either.

3.12. The Network of Terms searches specific parts of terminology sources

Priority: Should have

Users want to get as little noise as possible in the terms that the Network of Terms finds. For example, if a terminology source contains terms related to both persons and organizations, but a user is only interested in organizations, then he does not want to find persons. In that case, the user wants to be able to indicate in his search that he is only looking for organizations. Based on this information, the Network of Terms can then search specific parts of a terminology source rather than the whole source.

The Network of Terms can only search in specific parts of terminology sources if they support this. For example: a source can use concept schemes for grouping related terms (e.g. "organizations") or use types for expressing the nature of terms, e.g. a schema:Organization.

Discussion: this requirement need to be discussed further. For example: if a terminology source consists of four concept schemes, should the Network of Terms have knowledge of this? Or should the owner of the terminology source register four datasets in the Registry, one for each concept scheme? Another example: to what extent should the Network of Terms have knowledge of the types of terms, for example in order to distinguish a "person" from an "organization"?

3.13. The Network of Terms makes terms findable by searching with operators

Priority: Should have

Some users only vaguely know what term they are looking for. In that case they do not want to send a precise but a broad query to the Network of Terms. This can be done with wildcards. For example, the query *sword. The Network of Terms could then find terms such as "broadsword" or "long sword".

Other users know quite precisely what term they are looking for. In that case they want to ask the Network of Terms a fairly accurate question. This can be done with Boolean operators. For example, the query broadsword AND scimitar.

The Network of Terms can only search with operators if terminology sources support this. If a terminology source doesnât, then the Network of Terms cannot not provide this functionality for that source either.

Discussion: this requirement can be fulfilled in various ways. For example, the Network of Terms can offer its own uniform operators to users; the Network of Terms then translates these into operators supported by individual terminology sources. This is quite complicated, though. But we can also choose to not have the Network of Terms use uniform operators and simply use what the sources support. The consequence of this approach is that users may need to know different operators from different terminology sources.

3.14. The Network of Terms searches for terms with a specific status or parent term

Priority: Should have

Users want to find terms that best match their search query. This query contains one or more words, such as Zwaard or Haarlem. To prevent getting too many results, users want to complement their search query with additional criteria: "status" and "parent term". These criteria narrow the results.

With "status" a user can indicate that he only wants to find terms with this status. For example: only approved terms, no candidate terms.

With "parent term" a user can indicate that he only wants to find terms with this parent term. For example: only places that belong to "The Netherlands", not to "South Africa".

The Network of Terms can only search for terms with a certain status or parent term if the terminology sources support this. If a terminology source doesnât, then the Network of Terms cannot provide this functionality for that source either.

Discussion: should this requirement be split into two requirements, one for "status" and one for "parent term"?

3.15. The Network of Terms ensures that terms are ordered by relevance

Priority: Must have

Users want to find terms that best match their search query. The Network of Terms therefore ensures that matching terms are ordered by relevance: the terms that best answer the query are returned first.

The Network of Terms does not order the terms itself; the terminology sources do. If a terminology source does not support this functionality, the Network of Terms will return the terms in the default order of the terminology source.

The Network of Terms has no knowledge of the relevance ranking of terminology sources. It therefore cannot combine the terms from sources into one relevance-ranked result set. Instead, it groups the terms in the result set by source, respecting the order of each.

3.16. The Network of Terms returns exactly the information that the terminology sources provide, without any substantive changes

Priority: Must have

The Network of Terms allows users to search in terminology sources. It returns the matching terms directly to the users and does not change it. For example: it does not correct, enrich or re-sort terms - 'what you see is what you get'.

The Network of Terms does change the data structure with which terminology sources publish their terms. This change is necessary in order to get a uniform, source-independent structure (see requirement 1).

3.17. The Network of Terms monitors and displays the status of terminology sources

Priority: Must have

The Network of Terms queries terminology sources directly (see requirement 6). This means that there is a direct dependence between systems: if the system of a terminology source is, for whatever reason, unreachable, the Network of Terms cannot send the users' queries to the source and function properly. In order to quickly gain insight into problems, the Network of Terms keeps track of the status of terminology sources. The status makes clear whether or not the sources are reachable. If this is not the case, the application manager of the Network of Terms can take action, for example by contacting the owner of the source or by informing his users.

The Network of Terms provides a status page where the status of each terminology source is displayed. The status page is publicly accessible.

3.18. The application manager of the Network of Terms can add, edit and remove terminology sources

Priority: Should have

Terminology sources have their own protocols and data structures for querying and expressing terms. The Network of Terms knows about these, as well as how the information about terms is mapped into the uniform, source-independent data structure of the Network of Terms.

The application manager of the Network of Terms maintains the configuration about sources and mappings. He can add, edit and remove sources without the intervention of a software developer.

Discussion: this requirement needs further eloboration. For example: is it sufficient if an application manager can modify the configuration files of the Network of Terms? Or does he need a user interface for this?

3.19. The Network of Terms has developer-friendly documentation about its interfaces

Priority: Must have

Software developers of systems of users want to be able to implement the Network of Terms easily. Therefore the Network of Terms must have documentation that is accessible to developers. The documentation describes the interfaces of the Network of Terms, such as the query language, data format and metadata schemas. The documentation must be generated from the source code of the Network of Terms so that it is always up-to-date, e.g. with OpenAPI.

3.20. The Network of Terms publishes machine-readable information about its interfaces

Priority: Should have

The Network of Terms is primarily used by systems of users, not directly by users. Therefore the Network of Terms should not only have documentation for software developers (see requirement 19), but also publish machine-readable information about how its interfaces can be used by systems. This information makes the Network of Terms self-descriptive. The information makes clear, for example, which interfaces there are and which HTTP methods they support (such as POST or GET).

3.21. The Network of Terms has an availability of 99%

Priority: Must have

The Network of Terms is a generic facility that can be used at all times by all participants in the Dutch Digital Heritage Network. Therefore the Network of Terms must have a high availability of 99%.

Discussion: does this availability apply to all instances of the Network of Terms? Or can individual instances have a different availability?

Note that this availability only applies to the Network of Terms, not to the terminology sources that it queries; the owners of these sources are responsible for their own availability.
